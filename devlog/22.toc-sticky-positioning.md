# 22. TOC Sticky 定位优化

## 目标

修复 docs-toc 被 header 挡住的问题，实现 TOC 自管理定位。

## 问题分析

header 使用 `position: sticky` 固定在顶部后，TOC 的 `top: 0` 会被 header 遮挡。

### 为什么 TOC 无法完全自治

组件内部无法推导出：
- header 高度
- 顶部工具栏高度
- 安全区域
- sticky top

因为这些是布局上下文信息，浏览器没有提供 `element.stickyTop` 这样的 API。

## 解决方案

**职责分离**：sticky 定位由 outline 容器管理，TOC 只负责内容渲染。

### docs-page（outline 容器）

```css
.vvc-docs-page-outline {
  position: sticky;
  top: var(--vvc-toc-sticky-top, 0px);
  max-height: 100vh;
  overflow: auto;
}
```

### docs-toc（纯内容组件）

不再处理定位逻辑，只渲染目录内容。

## 用户自定义

```css
/* 方式1：直接设置 */
:root {
  --vvc-toc-sticky-top: 80px;
}

/* 方式2：关联 header 变量 */
:root {
  --vvc-toc-sticky-top: var(--header-height, 64px);
}
```

## 关键决策

1. **TOC 不依赖外部布局信息**：保持组件职责单一
2. **使用 CSS 变量 `--vvc-toc-sticky-top`**：用户可自定义，90% 场景开箱即用
3. **移除 container prop**：不再需要传递父级 DOM

## 参考

- VitePress、Docusaurus、Nextra 等框架通常由 Layout 层提供 offset
- 最稳的方案是读取 `getComputedStyle(el).top`，但前提是组件自己是 sticky
- 完全自治且 100% 正确做不到，这是信息论问题
