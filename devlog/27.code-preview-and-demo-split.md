# 27. 拆分 CodePreview 基础组件与 Demo 高层封装

## 目标

- 将原本 `demo` 的插槽式预览/源码容器抽成独立基础组件 `code-preview`。
- 让 `demo` 只负责 `preview + source[]` 这种注册式快捷 API。
- 不兼容旧的 `code-preview` API，直接以当前仓库需要的抽象为准。

## 当前标准

### CodePreview

- `code-preview` 是基础容器，只处理默认插槽预览区和 `#source` 插槽源码区。
- 它不解析 `preview` / `source` 属性，也不依赖 demos/sources 注册表。
- 源码区展开收起、样式容器和 `code-group` 的嵌套布局都由它负责。

### Demo

- `demo` 是 `code-preview` 的高层封装。
- `preview` 从 `demos` 里取组件；`source` 从 `sources` 里取源码 key 数组。
- `source` 为单项时渲染单个 `ProseCode`；多项时渲染 `CodeGroup`。
- `demo` 不再承载手写 slot 文档语义；手写预览/源码时应改用 `code-preview`。

### 文档站接入

- `ContentRenderer` 内置注册了 `code-preview`，文档和消费端都可以直接写 `:::code-preview`。
- `vike/pages/+content.ts` 不再需要为 `code-preview` 做额外组件注册。
- `demo.md` 只保留注册式用法。
- 新增 `code-preview.md` 承担原来 `demo` 的 slot / code-group 文档。

## 关键决策

- 不兼容旧 `code-preview` API：用户已经明确旧 API 不需要兼容，因此直接围绕当前抽象重建，避免背负历史接口噪音。
- `demo` 组合 `code-preview`，而不是继续自己维护同一套展开面板 DOM/CSS，减少重复实现。
- `code-preview` 的文案独立走 `codePreview.*` locale key，避免基础组件继续复用 `demo.*` 命名。
- `code-preview` 进入 `ContentRenderer` 内置映射，和 `code-group` 一样作为框架级基础组件提供。
