# 24. content 配置：Markdown 插件与组件注入

## 目标

让消费端通过 `+content.ts` 配置 Markdown 解析插件和自定义组件，并将这些组件透传到 `ContentRenderer`。

## 当前标准

消费端在 `vike/pages/+content.ts` 中声明 `content` 配置：

```ts
import highlight from 'vike-vue-content/comark/highlight'
import mermaid, { Mermaid } from '@comark/vue/plugins/mermaid'

export default {
  plugins: [highlight(), mermaid()],
  components: {
    mermaid: Mermaid,
  },
}
```

- `plugins`：Comark 解析插件数组，传给 `queryCollection` 用于服务端 Markdown 解析。
- `components`：标签名 → Vue 组件映射，透传到 `ContentRenderer` 的 `components` prop，用户传入优先于内置默认值（`a: ProseA`、`pre: ProseCode`）。

## 执行内容

### 1. `vike/package.json`

添加 `@comark/vue` 直接依赖，使 `+content.ts` 中的 `import mermaid, { Mermaid } from '@comark/vue/plugins/mermaid'` 在 pnpm 严格隔离下可解析。

### 2. `config/+config.ts`

`meta.content.env` 增加 `client: true`，使 `content` 配置（含 `components`）在客户端可用：

```ts
meta: {
  content: {
    env: {
      config: true,
      server: true,
      client: true,  // 新增
    },
  },
}
```

### 3. `shared/types/index.ts`

- 新增 `ContentComponents = Record<string, Component>` 类型。
- `ContentConfig` 增加 `components?: ContentComponents`。

### 4. `components/docs-page/src/index.vue`

通过 `usePageContext()` 读取 `pageContext.config.content?.components`，传给 `ContentRenderer`：

```ts
const pageContext = usePageContext()
const contentComponents = computed(() =>
  (pageContext.config as { content?: { components?: ContentComponents } }).content?.components
)
```

```html
<ContentRenderer :tree="contentTree" :components="contentComponents" />
```

### 5. `comark/highlight/index.ts`

导出从预配置实例改为函数，与 `mermaid()` 调用风格保持一致：

```ts
export default function highlightPlugin(options?: HighlightOptions) {
  return highlight({
    themes: { light: githubLight, dark: githubDark },
    ...options,
  })
}
```

## 关键事实

- `components` 不能通过 `data()` 传递：`pageContext.data` 是内置 `passToClient` 项，Vike 会尝试序列化整个 `data` 返回值，而 Vue 组件对象包含函数，无法序列化。
- 正确路径是通过 Vike config 系统传递：`meta.content.env.client: true` 使配置值通过 `pageContext.config` 在客户端可用，config 值不经过 `data()` 序列化路径。
- `content` 配置的 `plugins` 仍在服务端 `data()` hook 中读取（用于 `queryCollection`），`components` 则在客户端 `docs-page` 组件中从 config 读取——两者走不同路径。

## 当前边界

- `entry/` 是构建产物，需要重新运行 `bash build.sh` 才能同步 `config/+config.ts`、`comark/highlight/`、`components/docs-page/` 的变更。
- `prose-a` 组件（`components/prose-a/index.ts`）当前是 `Link` 的纯 re-export，尚未参与本次变更，下次构建会自动落盘到 `entry/components/prose-a/`。
