# 25. docs 插件 AST 化：支持 re-export 路由写法

## 目标

- 让 `+route.ts` 支持 `export { createDocsRoute as default } from 'vike-vue-content/docs/route'` 写法。
- 将 `docsRuntimeBasePlugin` 从正则匹配改为 AST 驱动，同时支持 JS/TS。

## 问题

`createDocsRoute()` 零参调用依赖编译期插件注入 base（如 `/zh-CN`）。旧插件用正则 `/\bcreateDocsRoute\s*\(\s*\)/g` 匹配零参调用并替换，但无法识别 re-export 模式：

```ts
// 旧插件无法处理 — 没有 createDocsRoute() 调用
export { createDocsRoute as default } from 'vike-vue-content/docs/route'
```

直接 re-export 导出的是工厂函数本身，Vike 在请求时以 `pageContext` 调用它，导致 `normalizeRoutePath()` 对 object 调用 `.replace()` 崩溃。

## 当前标准

### 消费端写法（三种均支持）

```ts
// 1. re-export（最简）
export { createDocsRoute as default } from 'vike-vue-content/docs/route'

// 2. 工厂调用
import { createDocsRoute } from 'vike-vue-content/docs/route'
export default createDocsRoute()

// 3. export { default }
export { default } from 'vike-vue-content/docs/route'
```

### 插件实现

`docs/plugin/plugin.ts` 使用 `oxc-parser`（Vite 生态，原生支持 JS/TS）：

1. **文件路径判断优先** — `isDocsRouteFile` / `isDocsPrerenderFile` 比 `code.includes()` 更快，非目标文件直接跳过。
2. **单次解析** — `parseSync` 生成 AST，`+route.ts` 文件只解析一次。
3. **Pass 1：顶层遍历** — 找到 `ExportNamedDeclaration` 中的 re-export 模式，替换为 `import + createDocsRoute("/zh-CN")`，保留其他 specifiers。
4. **Pass 2：Visitor 遍历** — 找到所有零参 `CallExpression`，在 `)` 前插入 docsBase 参数。
5. **右到左应用** — 按 start 位置降序应用编辑，避免位置偏移。

### 类型

使用 `oxc-parser` 导出的 AST 类型（re-export from `@oxc-project/types`）：
- `Program`、`ExportNamedDeclaration`、`ExportSpecifier`、`CallExpression`
- `ModuleExportName` — `IdentifierName | IdentifierReference | StringLiteral`，通过 `getName()` helper 统一取值（Identifier 用 `.name`，Literal 用 `.value`）
- `Visitor` — 类型安全的 AST 遍历

### createDocsRoute 函数签名

`docs/route/index.ts` 增加了 `pageContext` 重载作为安全网：

```ts
export function createDocsRoute(base?: string): DocsRoute
export function createDocsRoute(pageContext: PageContext): false | { routeParams: Record<string, string> }
```

当 Vike 直接调用工厂函数时（无插件环境），检测到 pageContext 对象会回退为创建默认路由并调用。

## 关键决策

- **AST 而非正则**：正则无法可靠处理 `export { a as default, b }` 等复合 specifier、注释干扰、TS 语法差异。oxc-parser 提供精确的 AST 节点位置。
- **oxc-parser 而非 es-module-lexer**：`es-module-lexer` 只能分析模块语法，无法访问 `CallExpression` 节点；`oxc-parser` 提供完整 AST + Visitor，且与 Vite 同生态。
- **编译期为主，runtime 为安全网**：插件负责注入 base，`createDocsRoute` 的 pageContext 重载只在无插件环境下兜底。
- **oxc-parser 和 @oxc-project/types 外部化**：原生 WASM 绑定不能打包进库产物，通过 `vite.config.ts` 的 `rolldownOptions.external` 排除。
